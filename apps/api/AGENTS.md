# AGENTS.md

Guidance for `apps/api`.

## Start Here

- Read the root `AGENTS.md` and `docs/technical-reference.md`.
- Read `docs/integrations.md` when touching provider flows, OAuth state, sync, or webhook handling.
- Use `apps/api/src/server.ts` and `apps/api/src/config/startup-env.ts` as the structural entry points for routes and runtime constraints.

## Local Rules

- Keep request validation at the route boundary and prefer shared schemas from `@triathlon/types`.
- Keep non-2xx responses in the existing problem-details format through the shared helpers.
- Protected routes live under `/api/*` and rely on `jwtAuth()` plus `extractClaims()`. Public routes are intentionally limited to `/health` and `/webhooks/*`.
- Keep startup env validation, `.env.example`, `packages/types/src/validation.ts`, and `.github/workflows/deploy.yml` aligned. `pnpm check:env-keys` verifies part of this contract.
- OAuth `returnTo` targets must remain allowlisted absolute `http(s)` URLs. Mobile uses `APP_LINK_URL`; custom schemes are intentionally rejected today.
- Do not hand-edit `dist` or `dist-deploy`. `dist-deploy` is generated by `pnpm --filter @triathlon/api build:deploy`.

## Docs to Update When Behavior Changes

- `docs/technical-reference.md`
- `docs/integrations.md`
- Root `README.md`
- `.env.example` and deploy workflow docs when env or Azure behavior changes

## Validation

- `pnpm --filter @triathlon/api test`
- `pnpm --filter @triathlon/api type-check`
- `pnpm check:env-keys` when env keys, deploy mapping, or startup validation changes
