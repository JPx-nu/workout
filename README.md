# triathlon-app

Monorepo for the current JPX triathlon product surface: a Next.js web cockpit, a Hono API, shared TypeScript packages, and a Flutter mobile companion.

## Current shipped scope

- Web athlete cockpit: auth, onboarding, dashboard, workouts, training calendar, AI Coach, 2D body map, settings, and live integrations control.
- API: AI chat/streaming, onboarding persistence, planned workouts, integrations, mobile health ingest, MCP bridge, and provider webhooks.
- Mobile companion: login, dashboard, workouts, training, coach, body map, and settings backed by the same API contracts for profile and integrations.

Not currently shipped as supported product surface:

- Garmin end-user availability
- HealthKit / Health Connect permission UX in the product
- 3D body map backed by live athlete data
- Self-serve export/delete-account flows in the UI
- Team relays / squad gamification

## Workspace map

- `apps/web` - Next.js 16 + React 19 web app on port `3100`
- `apps/api` - Hono + TypeScript API on port `8787`
- `packages/types` - shared contracts and Zod schemas
- `packages/core` - shared pure mapping/date/stats/strength logic
- `packages/api-client` - typed Hono RPC client scaffold
- `apps/mobile` - Flutter app outside pnpm/Turbo

## Quick start

1. `pnpm install`
2. Copy `.env.example` to `.env`
3. Fill in the required Supabase, Azure OpenAI, and provider credentials
4. Run `pnpm dev` or target a single workspace:
   - `pnpm --filter web dev`
   - `pnpm --filter @triathlon/api dev`

## Repo notes

- `apps/api/dist-deploy/` is generated by `pnpm --filter @triathlon/api build:deploy` and should not be committed.
- Web CSP `connect-src` is derived from `NEXT_PUBLIC_API_URL` and `NEXT_PUBLIC_WEB_URL` at build time.
- Web and mobile settings both consume the same `/api/integrations/status` contract and per-provider action routes.
- Cross-app data contracts should stay in `packages/types`; shared pure logic should stay in `packages/core`.
